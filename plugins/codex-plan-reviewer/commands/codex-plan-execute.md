사용자 요청: $ARGUMENTS

## Phase 0: Codex 사용 가능 여부 확인

Codex CLI 존재 여부를 확인합니다:

```bash
which codex || command -v codex
```

- 명령이 실패하면 사용자에게 `/codex:setup`을 실행하라고 안내하고 **즉시 중단**합니다.
- 성공하면 다음 Phase로 진행합니다.

이어서 `codex-companion.mjs` 스크립트의 경로를 확보합니다. 아래 순서로 탐색합니다:

1. `find ~/.claude/plugins -path "*/openai-codex/*/scripts/codex-companion.mjs" -type f 2>/dev/null | head -1`
2. 위 결과가 없으면 `find ~/.claude/plugins -name "codex-companion.mjs" -type f 2>/dev/null | head -1`

탐색된 경로를 이후 Phase에서 `CODEX_COMPANION` 변수로 사용합니다. 경로를 찾지 못하면 사용자에게 "`codex@openai-codex` 플러그인이 필요합니다. `/plugin install codex@openai-codex` 후 `/codex:setup`을 실행해주세요."라고 안내하고 중단합니다.

## Phase 1: 계획 입력 수집

`$ARGUMENTS`를 기반으로 계획 텍스트를 확보합니다:

- **파일 경로인 경우** (`/`, `./`, `~/`로 시작하거나 `.md`/`.txt`로 끝나는 경우): `Read` 도구로 파일을 읽습니다.
- **비어있는 경우**: `AskUserQuestion`으로 계획 텍스트를 입력하거나 계획 파일 경로를 알려달라고 요청합니다.
- **그 외**: `$ARGUMENTS` 텍스트를 계획으로 사용합니다.

### 검증

번호가 매겨진 구현 단계가 최소 1개 이상 포함되어야 합니다 (예: `1.`, `## 1.`, `**단계 1**`). 없으면 "구현 단계가 포함된 계획을 제공해주세요."라고 안내하고 중단합니다.

## Phase 2: 태스크 분할 및 의존성 분석

계획 텍스트에서 구현 단계를 추출하고 의존성을 분석합니다.

### 2-1. 태스크 추출

다음 패턴을 순서대로 매칭하여 번호 매겨진 단계를 추출합니다:

- `## N. 제목` (헤딩 스타일)
- `N. 제목` (번호 리스트)
- `**단계 N**: 설명` (볼드 스타일)

각 태스크에서 다음 정보를 추출합니다:

| 항목 | 설명 |
|------|------|
| 번호 | 원래 계획의 단계 번호 |
| 제목 | 태스크 이름 |
| 설명 | 태스크의 상세 작업 내용 |
| 영향받는 파일 | 수정/생성/삭제 대상 파일 목록 (계획에 명시된 경우) |

### 2-2. 의존성 추론

각 태스크를 분석하여 의존성을 추론합니다:

- 태스크 설명에서 이전 단계의 산출물(파일, 함수, 클래스 등)을 참조하는지 확인
- "단계 N에서 생성한", "위에서 만든" 등의 표현이 있으면 해당 단계에 대한 의존성 설정
- 명시적 의존성 표현이 없으면 독립 태스크로 분류 (단, 단계 번호 순서 자체를 의존성으로 간주하지 않음 — 내용 기반으로만 판단)

### 2-3. 웨이브 할당

의존성 그래프를 기반으로 토폴로지컬 웨이브를 할당합니다:

- **Wave 0**: 의존성이 없는 태스크
- **Wave N**: 의존하는 태스크가 모두 Wave 0~N-1에 속하는 태스크

### 2-4. 사용자 확인

추출 결과를 테이블로 표시합니다:

```markdown
## 추출된 태스크 목록

| # | 태스크 | 의존성 | 웨이브 | 영향받는 파일 |
|---|--------|--------|--------|--------------|
| 1 | 예시 태스크 A | 없음 | Wave 0 | file_a.ts |
| 2 | 예시 태스크 B | 1 | Wave 1 | file_b.ts |
| 3 | 예시 태스크 C | 없음 | Wave 0 | file_c.ts |
```

`AskUserQuestion`으로 확인받습니다:

1. **이대로 진행** (권장)
2. **의존성/웨이브 수정** — 사용자가 수정 사항을 텍스트로 입력
3. **취소**

옵션 2 선택 시 사용자의 수정 사항을 반영하고 테이블을 다시 표시합니다. 옵션 3 선택 시 중단합니다.

## Phase 3: 파일 충돌 검사

같은 웨이브 내에서 둘 이상의 태스크가 동일한 파일을 수정하는지 검사합니다. 영향받는 파일이 명시되지 않은 태스크는 충돌 검사 대상에서 제외합니다.

### 충돌이 발견되면

충돌 내용을 표시합니다:

```markdown
## 파일 충돌 감지

| 웨이브 | 충돌 파일 | 관련 태스크 |
|--------|----------|------------|
| Wave 0 | shared.ts | #1, #3 |
```

`AskUserQuestion`으로 해결 방안을 선택받습니다:

1. **충돌 태스크 병합** — 관련 태스크를 하나로 합침
2. **순차 실행으로 변경** — 충돌 태스크 중 하나를 다음 웨이브로 이동
3. **그대로 진행** — 충돌 위험을 감수하고 병렬 실행

### 충돌이 없으면

"파일 충돌 없음. 실행을 시작합니다."라고 알리고 다음 Phase로 진행합니다.

## Phase 4: 웨이브별 Codex 태스크 실행

현재 웨이브(Wave 0부터)의 태스크를 Codex 백그라운드 스레드로 실행합니다.

### 실행 방식

각 태스크에 대해 Bash 도구로 `codex-companion.mjs task --background --write`를 호출합니다. **같은 웨이브의 태스크들은 병렬로 Bash 호출을 실행합니다** (하나의 응답에서 여러 Bash 도구 호출).

```bash
node "<CODEX_COMPANION 경로>" task --background --write "<태스크 프롬프트>"
```

### 태스크 프롬프트 구성

각 태스크에 전달할 프롬프트는 다음 구조를 따릅니다:

```
## 프로젝트 컨텍스트
<전체 계획의 작업 개요 요약 (2-3문장)>

## 현재 태스크: <태스크 제목>
<태스크의 상세 설명>

## 영향받는 파일
<이 태스크가 수정/생성해야 할 파일 목록>

## 제약 조건
- 위에 명시된 파일만 수정하세요. 다른 파일은 건드리지 마세요.
- 기존 코드 패턴과 컨벤션을 따르세요.
- 완료 후 수정된 파일 목록과 변경 요약을 보고해주세요.
```

### 결과 기록

각 Bash 호출의 stdout에서 job ID를 파싱하여 기록합니다 (형식: `task-<timestamp>`). 실행된 태스크 목록을 표시합니다:

```markdown
## Wave N 실행 시작

| # | 태스크 | Job ID | 상태 |
|---|--------|--------|------|
| 1 | 태스크 A | task-1234567890 | 실행 중 |
| 3 | 태스크 C | task-1234567891 | 실행 중 |
```

## Phase 5: 진행 상황 모니터링 및 웨이브 진행

### 모니터링

`codex-companion.mjs status --json`으로 현재 웨이브의 모든 job 상태를 확인합니다:

```bash
node "<CODEX_COMPANION 경로>" status --json
```

현재 웨이브에서 실행한 job ID 목록을 기준으로, 해당 job들이 모두 `completed` 또는 `failed` 상태가 될 때까지 약 30초 간격으로 확인을 반복합니다. 이전 웨이브의 job은 무시합니다.

### 웨이브 완료 처리

웨이브의 모든 태스크가 완료되면 각 job의 결과를 수집합니다:

```bash
node "<CODEX_COMPANION 경로>" result <job-id> --json
```

결과에서 `touchedFiles`, `rawOutput`, `status`를 추출하여 진행 상황 테이블을 업데이트합니다.

### 실패 처리

- 실패한 태스크에 **의존하는** 후속 태스크는 **차단** (해당 웨이브에서 제외)
- 실패한 태스크에 **의존하지 않는** 후속 태스크는 정상 실행
- 실패 내용을 사용자에게 표시

### 다음 웨이브 진행

남은 웨이브가 있으면 Phase 4로 돌아가 다음 웨이브를 실행합니다. 차단된 태스크는 건너뜁니다.

## Phase 6: 결과 취합 및 최종 보고

모든 웨이브가 완료되면 최종 보고서를 출력합니다:

```markdown
## 실행 완료 보고

| # | 태스크 | 상태 | 수정된 파일 | 웨이브 |
|---|--------|------|------------|--------|
| 1 | 태스크 A | 완료 | file_a.ts | Wave 0 |
| 2 | 태스크 B | 완료 | file_b.ts | Wave 1 |
| 3 | 태스크 C | 실패 | - | Wave 0 |
```

### 실패한 태스크가 있는 경우

```markdown
### 실패한 태스크

| # | 태스크 | 실패 원인 |
|---|--------|----------|
| 3 | 태스크 C | <에러 메시지 요약> |

### 차단된 태스크 (실패 태스크에 의존)

| # | 태스크 | 차단 원인 |
|---|--------|----------|
| 5 | 태스크 E | 태스크 #3 실패 |
```

### 다음 단계 안내

```markdown
### 다음 단계
- 실패한 태스크 재시도: `/codex:rescue <태스크 설명>`
- 전체 변경사항 검토: 수정된 파일들을 확인하고 테스트를 실행하세요
```

모든 사용자 대면 메시지는 **한국어**로 출력합니다.
