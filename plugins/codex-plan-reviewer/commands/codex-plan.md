사용자 요청: $ARGUMENTS

## Phase 1: Codex 사용 가능 여부 확인

간단한 체크로 Codex CLI 존재 여부를 확인합니다:

```bash
which codex || command -v codex
```

- 명령이 실패하면 사용자에게 `/codex:setup`을 실행하라고 안내하고 **즉시 중단**합니다.
- 성공하면 다음 Phase로 진행합니다.

## Phase 2: 코드베이스 분석 및 계획 수립

`$ARGUMENTS`를 기반으로 **Explore 에이전트**(필요시 최대 3개 병렬)로 관련 코드를 탐색한 뒤, **Plan 에이전트**로 구현 계획을 작성합니다.

계획에 포함할 항목:

| 섹션 | 내용 |
|------|------|
| 작업 개요 | 무엇을, 왜 하는지 |
| 현재 코드베이스 분석 | 관련 파일, 기존 패턴, 재사용 가능한 코드 |
| 구현 단계 | 번호가 매겨진 단계별 작업 |
| 영향받는 파일 | 수정/생성/삭제 대상 파일 목록 |
| 위험 및 고려사항 | 잠재적 문제, 호환성, 성능 영향 |

계획 작성 후, 사용자에게 **"계획 수립 완료. Codex 리뷰를 요청합니다."** 라고 알립니다.

## Phase 3: Codex 리뷰 요청

Agent 도구로 `codex:codex-rescue` 서브에이전트를 실행하여 계획을 리뷰받습니다:

```
Agent({
  subagent_type: "codex:codex-rescue",
  prompt: "<계획 요약 또는 전문>\n\n위 구현 계획을 리뷰해주세요. 다음 관점에서 검토해주세요:\n1. 아키텍처 적합성\n2. 누락된 엣지케이스\n3. 더 나은 대안이 있는지\n4. 구현 순서의 적절성\n5. 잠재적 브레이킹 체인지\n\n코드를 수정하지 말고, 리뷰 피드백만 제공해주세요."
})
```

**계획 전달 기준**: 구현 단계가 10단계 이하이면 전문 전달, 초과하면 핵심 포인트로 요약하여 전달합니다. Codex의 응답은 그대로 보존합니다.

## Phase 4: 피드백 검증

Codex의 피드백을 **실제 코드베이스와 대조**하여 타당성을 검증합니다.

### 검증 원칙

- **명백히 유효한 피드백** (일반적 베스트 프랙티스 등): 추가 검증 없이 즉시 "유효" 판정
- **코드베이스 참조가 있는 피드백**: 해당 파일/함수가 실제 존재하는지 확인 (Read, Grep, Glob)
- **명백히 부정확한 피드백**: Phase 2 분석 결과와 대조하여 즉시 "무효" 판정

### 판정 기준

| 판정 | 의미 |
|------|------|
| **유효** | 피드백이 정확하며 계획에 반영해야 함 |
| **부분 유효** | 일부 맞으나 현재 컨텍스트에서는 수정 필요 |
| **무효** | 잘못된 가정에 기반하거나 현재 상태와 불일치 |

### 검증 결과 출력

```markdown
## Codex 피드백 검증 결과

| # | 피드백 요약 | 판정 | 근거 |
|---|------------|------|------|
```

## Phase 5: 최종 계획 업데이트

`AskUserQuestion`으로 반영 방식을 선택받습니다:

1. **유효한 피드백 모두 반영** (권장)
2. **피드백별 선택적 반영**
3. **원래 계획 유지**

- 옵션 1 또는 2 선택 시: 반영된 피드백과 변경 이력을 함께 표시하고, Phase 2와 동일한 구조로 최종 계획을 출력합니다.
- 옵션 3 선택 시: "원래 계획을 유지합니다."로 마무리합니다. 계획을 다시 출력하지 않습니다.

모든 사용자 대면 메시지는 **한국어**로 출력합니다.
